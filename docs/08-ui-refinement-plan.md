# UI改善計画（2026-07-22 ユーザーフィードバック一括対応）

実機運用フィードバック13件の整理と対応計画。**実装は本計画の承認後に着手する。**

対応順は「独立した小さい修正 → 構造をいじる修正 → 新機能」の順。
各項目に現状の根本原因（コードで確認済み）と対応方針・変更ファイル・検証方法を記す。

## A. 管理画面

### A-1. 会計履歴の明細が見づらい（文字を大きくはっきり）
- **現状**: 明細行が `fontSize: 13px` + `color: var(--text-2)`（半透明グレー）、
  小計等の内訳は `12px` + `var(--text-3)`（さらに薄い）。情報密度優先の設計が
  実店舗の照明・距離では読みにくい。
- **対応**: 明細行を `15px` + `var(--text)`（本文色）に、数量・金額は `700` ウェイト。
  内訳（小計/割引/税）は `13px` + `var(--text-2)` に引き上げ。日付・卓ヘッダーは現状維持
  （階層感は保つ）。
- **変更**: `components/history/CheckoutHistory.tsx` のみ。
- **検証**: dev サーバーでスクリーンショット比較。

### A-2. 通知音オンをデフォルトに
- **現状**: `store/useAppStore.ts` の初期値が `soundOn: false`。リロードごとにリセット
  （永続化なし）＝毎回手動でONにする必要があった。
- **対応**: 既定を ON に変更し、さらに **端末ローカル（localStorage）に永続化**する
  （リロードで OFF に戻ると「デフォルトON」の意味が薄いため。テーマモードと同じ方式）。
- **注意（仕様上の限界）**: ブラウザの自動再生ポリシーにより、ページを開いてから
  一度も画面に触れていない間は音が鳴らせない。厨房端末は通常何かしらタップするので
  実運用への影響は小さいが、「開いた直後の1件目だけ鳴らない」ことがあり得る。
- **変更**: `store/useAppStore.ts`（初期値+永続化）、必要なら `lib/` に小ヘルパー。

### A-3. 「切断をシミュレート」ボタンの必要性 → **廃止を推奨**
- **現状**: `toggleConnection` はローカル state を反転するだけの **M1（デモ期）の残骸**。
  実際の接続状態は `subscribeConnection`（Supabase Realtime のチャネル状態）が
  監視しており、本物の切断検知・オフラインバナーは既に機能している。
- **対応**: ボタンを削除。「接続中/オフライン」のステータス表示とオフライン警告バナーは
  維持。store の `toggleConnection` アクションも削除（参照が無くなるため）。
- **変更**: `components/kitchen/KitchenDisplay.tsx`、`store/useAppStore.ts`。

### A-4. 伝票内の項目に番号を振る
- **現状**: 厨房伝票の明細は品名+数量のみで、口頭確認時に指し示しにくい。
- **対応**: 伝票ごとに 1. 2. 3. と連番を表示（`tabular-nums`、`var(--text-2)` で控えめに）。
- **変更**: `components/kitchen/KitchenDisplay.tsx` の明細 map（index は既に取れている）。

### A-5. 注文反映のラグが大きい → **改善可能（原因3つを特定済み）**
- **原因① 人工的な800ms遅延（最大の要因・即除去可）**:
  `submitOrder` が `setTimeout(…, 800)` で送信全体を包んでいる。M1デモ期に
  「送信中…」の演出のため入れた遅延の残骸で、現在は害しかない。
  → **除去**。これだけで客側の体感が約1秒縮む。
- **原因② Realtime受信のデバウンス250ms**:
  `SupabaseSync` は変更通知を250msデバウンスしてから再取得する（orders と order_items の
  連続イベントを1回のfetchにまとめるため）。→ **150msへ短縮**（まとめ効果は維持しつつ
  100ms稼ぐ。それ以下はイベントが分裂して逆にfetch回数が増えるリスク）。
- **原因③ 構造的な遅延（今回そのまま）**: Edge Function往復＋Realtime配信＋再取得fetchで
  端末間反映は正味0.5〜1.5秒程度が物理限界。受信payloadから直接stateを組み立てれば
  さらに縮むが、整合性リスクの割に得る効果が小さいため今回は見送り（必要になれば別途）。
- **変更**: `store/useAppStore.ts`（800ms除去）、`components/SupabaseSync.tsx`（250→150ms）。
- **検証**: 2タブ（客画面/厨房）で注文→反映の時間を目視計測（前後比較）。

### A-6. ホーム画面追加バナーが一回しか表示されない
- **現状**: 閉じると **14日間再表示しない**設計にしていた（localStorage記録）。
  「後で入れよう」と閉じた場合に二度と出てこないように見える。
- **対応**: 抑制を「**そのブラウザセッションの間だけ**」（sessionStorage）に変更。
  タブを閉じて開き直せば再度表示される。**インストール済みの検知**
  （`display-mode: standalone` + `appinstalled` イベント）を追加し、
  インストール後は一切出さない。
- **変更**: `components/PwaController.tsx`。

## B. 客用注文画面

### B-1. トップエリアがヘッダー写真と被って不細工／縦に縮小したい
- **現状**: 浮遊ガラスヘッダーが2行構成（店名+卓名 ／ 右に「履歴」「スタッフ呼出」が縦積み）
  で高さ約80px。スクロール領域の上部スペーサーが88pxのため、ヘッダー写真がガラスの
  下に食い込んで見える。
- **対応**: **1行構成に再設計**。左=卓名（店名は上に小さく維持）、右=「履歴」「呼出」を
  横並び。高さを約56pxに圧縮し、スペーサーも同期して縮小（ヘッダー写真が被らない値に調整）。
- **変更**: `components/customer/CustomerOrder.tsx`（ヘッダー部+スペーサー2箇所）。
- **検証**: ヘッダー写真ON/OFF両方でスクリーンショット確認（写真が無い店でも崩れないこと）。

### B-2. 下の注文・合計金額エリアも少し縦縮小
- **現状**: カートバーは注意書き＋点数/合計＋ボタン＋税注記の4段で約128px。
- **対応**: 注意書き「変更・キャンセルは…」を点数行と統合するか文字サイズ・余白を圧縮し、
  全体で約15〜20px縮小。ボタンの押しやすさ（高さ44px以上）は維持。
  下部スペーサー(136px)も同期調整。
- **変更**: 同上ファイル。

### B-3. スクロールバーの有無で要素が左右にずれる → 常時表示
- **現状**: メニュー一覧が `overflowY: auto` のため、品数の少ないカテゴリでは
  スクロールバーが消え、コンテンツ幅が変わって要素が左右に動く
  （クラシックスクロールバーの環境=Windows等で顕著）。
- **対応**: スクロール領域に `overflow-y: scroll` + `scrollbar-gutter: stable` を指定し、
  バー領域を常時確保。管理画面（document スクロール）にも `html { scrollbar-gutter: stable }`
  を適用して同種のずれを予防。
- **変更**: `components/customer/CustomerOrder.tsx`、`app/globals.css`。

### B-4. オプション選択の文言変更
- **現状**: 「オプションを選んでください（いくつでも選べます）」
- **対応**: 「オプションを選んでください（**複数選択できます**）」に変更。
- **変更**: `components/ui/OptionSheet.tsx`（客・スタッフ代理の両方がこの1ファイルを共有）。

### B-5. （管理画面）オプション追加ウィンドウが画面中央に来ない → **原因特定済み**
- **現状**: メニュー管理のオプション編集モーダルは `position: fixed` で中央配置している
  **つもり**だが、祖先のガラスカードに `backdrop-filter` があるため、CSS仕様により
  fixed の基準がビューポートではなく**そのカード**になる（backdrop-filter は fixed 要素の
  containing block を作る）。縦に長いメニュー一覧カードの中央=スクロール位置によっては
  画面外、という報告どおりの症状になる。
- **対応**: モーダルを **React Portal（`createPortal(…, document.body)`）** で body 直下に
  描画し、ガラスカードの影響圏から出す。今後モーダルを glass カード内から開く場合は
  必ずPortalを使う、という規約もコードコメントに残す。
- **変更**: `components/menu/MenuManagement.tsx`（ItemOptionsButton のモーダル部分）。
- **検証**: メニュー一覧を最下部までスクロールした状態で「オプション」を開き、
  ビューポート中央に出ることを確認。

## C. 共通（新機能）

### C-1. 左右スワイプでカテゴリ移動（客画面・管理画面）
- **対象**: 客画面のメニュー一覧、管理画面の代理注文グリッド、メニュー管理一覧。
  それぞれのカテゴリチップ（customerCat / proxyCat / adminCat）を
  スワイプで隣のカテゴリへ切り替える（「すべて」を含む並び順で循環はしない・端で停止）。
- **実装**: 共通フック `lib/useSwipeCategory.ts`（新規）。touchstart/touchend の差分で
  水平±50px超・垂直優位でないときのみ発火。
- **競合対策（重要）**:
  - 縦スクロールとの誤爆: 水平移動量 > 垂直移動量×1.5 のときだけスワイプ判定。
  - **メニュー管理の行は横スクロールする**（`.menu-rows`）ため、タッチ開始点が
    「横にスクロール可能な要素の内側」の場合はスワイプ判定を無効化
    （`scrollWidth > clientWidth` の祖先を辿って自動判定）。
  - カテゴリ切替時はチップ行の選択位置まで自動スクロール（今どこかを見失わないように）。
- **変更**: 新規フック + `CustomerOrder.tsx` / `StaffCheckout.tsx` / `MenuManagement.tsx`。
- **検証**: 実機タッチでの誤爆確認が必須（縦スクロール中に切り替わらないこと、
  メニュー管理の行横スクロールが従来通り動くこと）。
